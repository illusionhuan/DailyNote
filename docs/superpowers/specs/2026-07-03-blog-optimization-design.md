## 1. 背景与目标

### 1.1 当前问题

1. **图片文件过大**：总计 51MB，严重影响页面加载速度
   - 1.png: 13MB
   - 2.jpg: 13MB
   - landscape5.jpg: 16MB
   - landscape4.jpg: 3.6MB
   - landscape3.jpg: 2.6MB

2. **配置路径不一致**：
   - `avatar.img: /images/9-tuya.jpg`（相对路径）
   - `archive_img: /DailyNote/images/landscape1.jpg`（完整路径）
   - 维护困难，切换自定义域名时需要修改多处硬编码路径

3. **热文历史归档功能不一致**：
   - 需求文档说"已移除（功能冗余）"
   - 但 `fetch_trending.py` 还在保存历史快照
   - 存在代码冗余和功能不一致

4. **性能优化空间**：
   - 前端性能可进一步优化
   - 构建配置可优化
   - SEO 可增强

### 1.2 优化目标

1. **图片优化**：将 51MB 压缩至 10MB 以内
2. **配置路径统一**：所有路径使用完整路径格式
3. **删除热文历史归档**：完全删除历史数据和相关代码
4. **性能优化**：前端性能、构建优化、SEO 增强

### 1.3 成功标准

- 图片总大小降至 10MB 以内
- 所有配置路径统一为完整路径格式
- 热文历史归档功能完全删除
- 页面加载速度提升 50% 以上
- 构建时间减少 20% 以上

## 2. 分阶段优化方案

### 2.1 阶段一：图片优化（最高优先级）

**目标：** 将 51MB 的图片压缩到 10MB 以内

**具体步骤：**

1. **创建压缩脚本**
   - 文件：`blog/scripts/compress_images.py`
   - 功能：批量压缩图片，保留原始格式
   - 依赖：Pillow 库

2. **压缩图片**
   - 1.png (13MB) → 压缩至 ~2MB
   - 2.jpg (13MB) → 压缩至 ~2MB
   - landscape5.jpg (16MB) → 压缩至 ~3MB
   - landscape4.jpg (3.6MB) → 压缩至 ~1MB
   - landscape3.jpg (2.6MB) → 压缩至 ~1MB

3. **备份原始图片**
   - 备份目录：`blog/source/images/backup/`
   - 保留原始文件，便于回滚

4. **验证压缩效果**
   - 对比压缩前后的图片质量
   - 测试页面加载速度
   - 验证轮播功能正常

**预期效果：**
- 图片总大小从 51MB 降至 ~10MB
- 页面加载速度提升 5-10 倍
- 移动端用户体验显著改善

**验证方法：**
- 使用 `du -sh` 命令检查图片目录大小
- 使用浏览器开发者工具测试加载速度
- 手动测试轮播功能

**回滚方案：**
- 从备份目录恢复原始图片
- 重新构建部署
