## OSI 七层模型和 TCP/IP 四层模型

### OSI 七层模型是什么？

**OSI 七层模型**是国际标准化组织提出的网络分层模型，从上到下依次为：

| 层 | 名称 | 作用 |
|----|------|------|
| 7 | **应用层** | 为应用程序提供网络服务（HTTP、FTP、SMTP） |
| 6 | **表示层** | 数据格式转换、加密解密、压缩解压缩 |
| 5 | **会话层** | 建立、管理和终止会话 |
| 4 | **传输层** | 提供端到端的数据传输服务（TCP、UDP） |
| 3 | **网络层** | 负责路由选择和逻辑寻址（IP、ARP、ICMP） |
| 2 | **数据链路层** | 将 IP 数据报组装成帧，在相邻节点间传送 |
| 1 | **物理层** | 实现比特流的透明传送 |

OSI 概念清楚但**过于复杂不实用**，有些功能在多层中重复出现。

### TCP/IP 四层模型是什么？

TCP/IP 四层模型是目前广泛采用的模型：

1. **应用层**：对应 OSI 上三层，提供应用程序间信息交换服务
2. **传输层**：负责两台终端设备进程之间的通用数据传输（TCP/UDP）
3. **网络层**：负责路由选择和分组转发（IP、ARP、NAT、ICMP）
4. **网络接口层**：对应 OSI 的数据链路层和物理层

### 为什么网络要分层？

1. **各层之间相互独立**：每层只需知道如何调用下层提供的功能
2. **提高整体灵活性**：每层可以使用最适合的技术实现
3. **大问题化小**：将复杂的网络问题分解为多个边界清晰的小问题

## TCP 与 UDP 的区别

| 对比项 | TCP | UDP |
|--------|-----|-----|
| 是否面向连接 | **是** | 否 |
| 是否可靠 | **是** | 否 |
| 是否有状态 | **是** | 否 |
| 传输效率 | 较慢 | **较快** |
| 传输形式 | 字节流 | 数据报文段 |
| 首部开销 | 20~60 字节 | **8 字节** |
| 是否支持广播/多播 | 否 | **是** |

**核心区别**：TCP 通过三次握手建立连接，具备确认、窗口、重传、拥塞控制机制，保证数据**无差错、不丢失、不重复、按序到达**。

**选择建议**：
- **UDP**：即时通信（语音、视频、直播）
- **TCP**：文件传输、邮件收发、远程登录

## TCP 三次握手和四次挥手

### 为什么要三次握手？

三次握手的目的是**确认双方的发送能力和接收能力都正常**：

1. **第一次**：客户端发送 SYN → 服务端确认客户端发送能力正常
2. **第二次**：服务端回复 SYN + ACK → 客户端确认服务端收发能力正常
3. **第三次**：客户端发送 ACK → 服务端确认客户端接收能力正常

还能**防止已失效的连接请求**突然到达服务端导致资源浪费。

### 为什么第二次还要传回 SYN？

ACK 确认客户端的 SYN 报文，SYN 是**服务端向客户端发起建立连接的请求**，让客户端确认服务端的发送能力是否正常。

### 为什么要四次挥手？

TCP 是**全双工通信**，关闭连接时双方都需要单独关闭：

1. 客户端发 FIN → 表示客户端没有数据要发了
2. 服务端回 ACK → 表示已收到，但可能还有数据要发
3. 服务端发 FIN → 表示服务端也没有数据要发了
4. 客户端回 ACK → 等待 2*MSL 后关闭连接

### 为什么第四次挥手要等待 2*MSL？

1. 确保最后一个 ACK 能到达服务端（超时会重发 FIN）
2. 让本次连接的所有报文在网络中消失，防止影响新连接

## TCP 如何保证传输的可靠性？

- **序列号和确认应答**：每个数据包有序列号，接收方通过 ACK 确认
- **超时重传**：规定时间内未收到 ACK，重新发送
- **滑动窗口**：接收方通过窗口大小告知发送方自己的接收能力
- **拥塞控制**：慢启动、拥塞避免、快重传、快恢复
- **校验和**：检测数据是否在传输中发生变化

## 从输入 URL 到页面展示到底发生了什么？

1. **DNS 解析**：浏览器缓存 → 操作系统缓存 → hosts 文件 → 本地 DNS → 根 DNS → 顶级域名 DNS → 权威 DNS
2. **TCP 连接**：三次握手建立连接（HTTPS 还需 TLS 握手）
3. **发送 HTTP 请求**：构建请求报文发送给服务器
4. **服务器处理并返回响应**：处理请求返回 HTTP 响应报文
5. **浏览器解析渲染页面**：HTML 构建 DOM 树 → CSS 构建 CSSOM 树 → 合并渲染树 → 布局 → 绘制
6. **连接结束**：四次挥手断开 TCP 连接（Keep-Alive 则保持）

## HTTP 状态码有哪些？

| 类别 | 含义 | 常见状态码 |
|------|------|-----------|
| **1xx** | 信息性 | 100 Continue |
| **2xx** | 成功 | 200 OK、204 No Content、206 Partial Content |
| **3xx** | 重定向 | 301 永久重定向、302 临时重定向、304 Not Modified |
| **4xx** | 客户端错误 | 400 Bad Request、401 Unauthorized、403 Forbidden、404 Not Found |
| **5xx** | 服务端错误 | 500 Internal Server Error、502 Bad Gateway、503 Service Unavailable |

## HTTP 和 HTTPS 有什么区别？

| 对比项 | HTTP | HTTPS |
|--------|------|-------|
| 端口号 | 80 | **443** |
| URL 前缀 | http:// | https:// |
| 安全性 | **明文传输** | **加密传输** |
| 资源消耗 | 较少 | 较多（加解密开销） |
| 证书 | 不需要 | 需要 CA 证书 |

**HTTPS 加密原理**：运行在 SSL/TLS 之上，使用**对称加密**加密传输内容，对称加密的密钥通过**非对称加密**（服务器证书）进行交换。

## HTTP 1.0 和 HTTP 1.1 有什么区别？

- **连接方式**：1.0 短连接 vs 1.1 **长连接（Keep-Alive）**
- **状态响应码**：1.1 新增大量状态码
- **缓存处理**：1.0 用 If-Modified-Since/Expires；1.1 引入 **ETag**、If-None-Match 等
- **带宽优化**：1.1 引入 **Range** 头域支持范围请求（206）
- **Host 头**：1.1 请求头加入 Host 字段，支持虚拟主机

## HTTP 无状态如何保存用户状态？

HTTP 是无状态协议，通过 **Session 机制**在服务端记录用户状态。服务端为特定用户创建 Session，通过 Session ID 标识用户。大部分情况下通过在 **Cookie 中附加 Session ID** 来跟踪。Cookie 被禁用时，可使用 **URL 重写**将 Session ID 附加在 URL 路径后面。

## URI 和 URL 的区别？

- **URI**（统一资源标识符）：唯一标识一个资源，侧重**标识**（如身份证号）
- **URL**（统一资源定位符）：唯一标识资源并提供定位方法，侧重**定位**（如家庭住址）

URL 是 URI 的子集，不仅标识资源，还提供定位该资源的方法（协议、主机、端口、路径）。
