## 为什么需要建表规范？

糟糕的表设计会导致：字段类型不合理浪费存储、索引设计不当导致慢查询、命名混乱增加维护成本。建表规范是数据库性能和可维护性的**基石**。

## 命名规范

### 表名规范

| 规则 | 说明 | 示例 |
|------|------|------|
| 小写字母 + 下划线 | 统一风格，避免大小写敏感问题 | `user_order` |
| 业务前缀 | 区分不同业务模块 | `pay_order`、`log_login` |
| 单数形式 | 表名用单数 | `user` 而非 `users` |
| 禁止保留字 | 避免与 MySQL 关键字冲突 | `order` → `t_order` 或 `user_order` |
| 长度不超过 30 字符 | 便于阅读和工具兼容 | — |

### 字段名规范

| 规则 | 说明 | 示例 |
|------|------|------|
| 小写字母 + 下划线 | 与表名风格一致 | `user_name` |
| 主键统一为 `id` | 简洁明确 | `id BIGINT PRIMARY KEY` |
| 布尔字段 `is_` 前缀 | 语义清晰 | `is_deleted`、`is_active` |
| 时间字段 `_at` 或 `_time` 后缀 | 区分日期和时间戳 | `created_at`、`login_time` |
| 外键字段与关联表主键同名 | 减少歧义 | `orders.user_id` 关联 `users.id` |

### 索引命名规范

```sql
-- 主键：不需要命名（自动 PRIMARY）
-- 唯一索引：uk_表名_字段名
ALTER TABLE users ADD UNIQUE INDEX uk_users_email (email);

-- 普通索引：idx_表名_字段名
ALTER TABLE orders ADD INDEX idx_orders_user_id (user_id);

-- 联合索引：idx_表名_字段1_字段2
ALTER TABLE orders ADD INDEX idx_orders_status_created (status, created_at);
```

## 字段类型选择

### 整数类型

| 类型 | 字节 | 范围（有符号） | 使用场景 |
|------|------|----------------|----------|
| TINYINT | 1 | -128 ~ 127 | 状态字段、布尔值 |
| SMALLINT | 2 | -32768 ~ 32767 | 小范围数值 |
| MEDIUMINT | 3 | -8388608 ~ 8388607 | 中等范围 |
| INT | 4 | -21亿 ~ 21亿 | 一般自增主键 |
| BIGINT | 8 | 极大 | **推荐主键类型** |

**最佳实践**：
- 主键统一使用 `BIGINT`，避免数据量增长后不够用
- 状态、类型等枚举值用 `TINYINT`，不要用 `VARCHAR`
- 金额用 `DECIMAL`，**绝不用** `FLOAT`/`DOUBLE`

### 字符串类型

| 类型 | 最大长度 | 存储方式 | 适用场景 |
|------|----------|----------|----------|
| CHAR | 255 字节 | 定长 | 长度固定的字段（如手机号、MD5 值） |
| VARCHAR | 65535 字节 | 变长 | 长度可变的短文本（如用户名、标题） |
| TEXT | 65535 字节 | 额外存储 | 长文本（如文章内容、评论） |

**最佳实践**：
- 能用 `VARCHAR` 就不用 `TEXT`（TEXT 不能有默认值，排序性能差）
- `VARCHAR` 长度按实际最大长度设置，不要统一 `VARCHAR(255)`（影响内存分配）
- 超长文本考虑拆表或存对象存储（S3/OSS）

### 时间类型

| 类型 | 格式 | 范围 | 建议 |
|------|------|------|------|
| DATE | YYYY-MM-DD | 1000-01-01 ~ 9999-12-31 | 只需日期时 |
| DATETIME | YYYY-MM-DD HH:MM:SS | 1000-01-01 ~ 9999-12-31 | **推荐** |
| TIMESTAMP | YYYY-MM-DD HH:MM:SS | 1970-01-01 ~ 2038-01-19 | 2038 年问题，慎用 |

**最佳实践**：
- 推荐 `DATETIME`，范围更大，不受 2038 问题影响
- `TIMESTAMP` 占 4 字节，`DATETIME` 占 8 字节，但在现代硬件上差异可忽略
- `created_at` 和 `updated_at` 是标配字段

### 金额类型

```sql
-- 错误：浮点数精度丢失
price FLOAT;         -- 0.1 + 0.2 ≠ 0.3

-- 正确：定点数
price DECIMAL(10,2); -- 最大 99999999.99，精确到分
```

**DECIMAL(M,D)**：M 是总位数，D 是小数位数。金融场景使用 `DECIMAL` 或**整数存分**（如 100 元存为 10000）。

## 字段设计规范

### NOT NULL 与默认值

```sql
-- 推荐：所有字段尽量 NOT NULL + 默认值
CREATE TABLE users (
    id BIGINT NOT NULL AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL DEFAULT '',
    email VARCHAR(100) NOT NULL DEFAULT '',
    age TINYINT NOT NULL DEFAULT 0,
    status TINYINT NOT NULL DEFAULT 1,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (id)
);
```

**为什么尽量 NOT NULL？**
1. NULL 参与的比较和运算需要特殊处理（`IS NULL`、`IS NOT NULL`）
2. NULL 列使索引更复杂，占用更多空间
3. NULL 值在 COUNT、SUM 等聚合函数中的行为容易出错
4. 联合索引中包含 NULL 列会影响索引效率

### 主键设计

```sql
-- 推荐：自增 BIGINT 主键
id BIGINT NOT NULL AUTO_INCREMENT PRIMARY KEY

-- 分布式场景：雪花算法生成 ID
id BIGINT NOT NULL PRIMARY KEY  -- 应用层生成 ID
```

**主键设计原则**：
1. **自增**：顺序插入，减少页分裂，写入性能最好
2. **BIGINT**：4 字节 INT 约 21 亿，可能不够用
3. **无业务含义**：避免业务变更导致主键需要修改
4. **不可修改**：主键一旦确定，永远不改

**反模式**：
- 使用 UUID 作为主键（随机写入，导致页分裂，索引空间大）
- 使用有业务含义的字段作为主键（如身份证号）

### 逻辑删除 vs 物理删除

```sql
-- 方案1：逻辑删除（推荐大多数业务场景）
is_deleted TINYINT NOT NULL DEFAULT 0  -- 0=正常, 1=已删除
-- 查询时必须加 WHERE is_deleted = 0

-- 方案2：物理删除（日志、临时数据等）
-- 直接 DELETE，数据不可恢复
```

**选择建议**：
- 需要数据审计、关联数据完整性的用**逻辑删除**
- 日志类、临时数据、无关联关系的用**物理删除**
- 逻辑删除时，`is_deleted` 字段需要加入唯一索引

### 枚举值字段

```sql
-- 反模式：用 ENUM 类型
status ENUM('pending', 'paid', 'shipped', 'completed')

-- 推荐：TINYINT + 注释
status TINYINT NOT NULL DEFAULT 0 COMMENT '0=待支付, 1=已支付, 2=已发货, 3=已完成'
```

**为什么不用 ENUM？**
1. ENUM 的值列表变更需要 `ALTER TABLE`（在线 DDL 也有锁）
2. ENUM 内部存储为整数，但与 TINYINT 相比无优势
3. ENUM 的值在不同环境可能出现排序不一致

## 索引设计规范

### 索引设计原则

| 原则 | 说明 |
|------|------|
| **最左前缀** | 联合索引遵循最左匹配，把区分度高的列放前面 |
| **覆盖索引** | 索引包含查询所需的所有列，避免回表 |
| **索引数量控制** | 单表索引不超过 5 个，过多影响写入性能 |
| **避免冗余索引** | (a,b) 已经覆盖 (a)，不需要再建 (a) |
| **避免重复索引** | PRIMARY KEY(id) 上再建 INDEX(id) 是多余的 |

### 联合索引设计

```sql
-- 场景：经常按 status 和 created_at 查询
-- 高区分度在前
ALTER TABLE orders ADD INDEX idx_status_created (status, created_at);

-- 查询可以利用索引
SELECT * FROM orders WHERE status = 1;                          -- ✅ 使用索引
SELECT * FROM orders WHERE status = 1 AND created_at > '2026-01-01';  -- ✅ 使用索引
SELECT * FROM orders WHERE created_at > '2026-01-01';           -- ❌ 无法使用索引
```

**联合索引字段顺序**：**等值查询的列 > 范围查询的列 > 排序列**

```sql
-- 频繁查询：WHERE status = 1 AND amount > 100 ORDER BY created_at
-- 最佳索引
ALTER TABLE orders ADD INDEX idx_status_amount_created (status, amount, created_at);
-- status 等值 → amount 范围 → created_at 排序
```

### 索引失效场景

```sql
-- 1. 索引列使用函数
WHERE YEAR(created_at) = 2026           -- ❌ 索引失效
WHERE created_at >= '2026-01-01'        -- ✅ 走索引

-- 2. 隐式类型转换
WHERE phone = 13800138000               -- ❌ phone 是 VARCHAR，类型不匹配
WHERE phone = '13800138000'             -- ✅

-- 3. 左模糊查询
WHERE name LIKE '%张'                   -- ❌ 索引失效
WHERE name LIKE '张%'                   -- ✅ 走索引

-- 4. OR 连接非索引列
WHERE user_id = 1 OR remark = 'test'    -- ❌ remark 无索引则全表扫描

-- 5. NOT IN / NOT EXISTS / !=
WHERE status != 1                       -- ❌ 可能不走索引（取决于数据分布）
WHERE status IN (0, 2, 3)              -- ✅ 改写为 IN

-- 6. 联合索引不满足最左前缀
-- 索引 (a, b, c)
WHERE b = 1 AND c = 2                  -- ❌ 跳过了 a
WHERE a = 1 AND c = 2                  -- ❌ 跳过了 b（MySQL 8.0 可能使用索引跳跃扫描）
```

### 前缀索引

```sql
-- 对长字符串字段使用前缀索引，节省空间
ALTER TABLE users ADD INDEX idx_email_prefix (email(10));

-- 选择合适的前缀长度
SELECT
    COUNT(DISTINCT LEFT(email, 5)) / COUNT(*) AS sel5,
    COUNT(DISTINCT LEFT(email, 10)) / COUNT(*) AS sel10,
    COUNT(DISTINCT LEFT(email, 15)) / COUNT(*) AS sel15,
    COUNT(DISTINCT email) / COUNT(*) AS sel_full
FROM users;
-- 选择区分度接近 sel_full 的最短前缀
```

**前缀索引的限制**：无法用于 ORDER BY 和覆盖索引。

## 表结构规范

### 必备字段

```sql
CREATE TABLE example (
    id          BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    is_deleted  TINYINT NOT NULL DEFAULT 0 COMMENT '逻辑删除: 0=正常, 1=已删除',
    PRIMARY KEY (id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='示例表';
```

### 字符集与排序规则

```sql
-- 推荐：utf8mb4（支持 emoji 和特殊字符）
CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci

-- 不推荐：utf8（只支持 3 字节字符，不支持 emoji）
```

**统一使用 `utf8mb4`**，避免因字符集不一致导致 JOIN 性能下降或数据截断。

### 存储引擎

```sql
-- 统一使用 InnoDB
ENGINE=InnoDB
```

MySQL 5.5.5 起 InnoDB 是默认引擎，支持事务、行级锁、MVCC、崩溃恢复。

### 行格式

```sql
-- 推荐 DYNAMIC（MySQL 5.7+ 默认）
ROW_FORMAT=DYNAMIC

-- 或在 my.cnf 中全局配置
-- innodb_row_format = DYNAMIC
```

`DYNAMIC` 格式对大字段（VARCHAR、TEXT、BLOB）的存储更高效，长字段只存储 20 字节指针。

## 在线 DDL 注意事项

```sql
-- MySQL 5.6+ 支持 Online DDL
ALTER TABLE users ADD INDEX idx_email (email), ALGORITHM=INPLACE, LOCK=NONE;

-- 查看 DDL 进度
SHOW PROCESSLIST;

-- 大表 DDL 推荐使用 pt-online-schema-change
pt-online-schema-change --alter "ADD INDEX idx_email (email)" \
    D=test,t=users --execute
```

**DDL 风险**：
1. `ALTER TABLE` 在大表上执行可能锁表数分钟
2. 推荐使用 **pt-online-schema-change** 或 **gh-ost** 做在线表结构变更
3. 避免在业务高峰期执行 DDL

## 数据库设计范式与反范式

### 三大范式

| 范式 | 要求 | 示例 |
|------|------|------|
| **1NF** | 字段不可再分 | 手机号和座机号分开存，不要合在一个字段 |
| **2NF** | 非主键字段完全依赖主键 | 拆表消除部分依赖 |
| **3NF** | 非主键字段不传递依赖主键 | 订单表不存用户名称，只存 user_id |

### 适度反范式

```sql
-- 范式设计：需要 JOIN
SELECT o.*, u.username FROM orders o JOIN users u ON o.user_id = u.id;

-- 反范式设计：冗余 username，避免 JOIN
-- orders 表增加 username 字段
SELECT * FROM orders;  -- 直接能拿到 username
```

**反范式的适用场景**：
1. 频繁的多表 JOIN 查询
2. 读多写少的场景
3. 数据一致性要求不高的场景（如用户昵称）

**注意**：冗余字段需要在更新时同步维护，增加写入复杂度。

## 建表 Checklist

| 检查项 | 要求 |
|--------|------|
| 表名 | 小写 + 下划线，有业务前缀 |
| 主键 | BIGINT 自增，无业务含义 |
| 字符集 | utf8mb4 |
| 存储引擎 | InnoDB |
| NOT NULL | 所有字段尽量 NOT NULL + 默认值 |
| 时间字段 | created_at + updated_at |
| 逻辑删除 | is_deleted（需要时） |
| 索引数量 | 单表不超过 5 个 |
| 索引命名 | idx_表名_字段 / uk_表名_字段 |
| 注释 | 表和字段必须有 COMMENT |
| 金额 | DECIMAL，不用 FLOAT |
| 状态字段 | TINYINT + COMMENT |
