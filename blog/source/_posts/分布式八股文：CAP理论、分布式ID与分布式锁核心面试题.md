## 理论&算法&协议

### 什么是 CAP 理论？

CAP 指分布式系统的三个特性：

- **一致性（Consistency）**：所有节点访问同一份最新数据副本
- **可用性（Availability）**：非故障节点在合理时间内返回合理响应
- **分区容错性（Partition Tolerance）**：出现网络分区时仍能对外提供服务

**核心结论**：分区容错性 P 是必须满足的，在此基础上只能选择 **CP**（强一致性）或 **AP**（高可用性），不可能选择 CA。

- **CP 架构**：ZooKeeper、HBase
- **AP 架构**：Cassandra、Eureka
- **Nacos** 同时支持 CP 和 AP

### 什么是 BASE 理论？

BASE 是对 CAP 中 AP 方案的补充：

- **Basically Available（基本可用）**：系统出现故障时允许损失部分可用性
- **Soft-state（软状态）**：允许系统中间状态存在
- **Eventually Consistent（最终一致性）**：所有数据副本最终达到一致

核心思想：即使无法做到强一致性，也可以根据业务特点达到**最终一致性**。

### Paxos 和 Raft 算法

**Paxos**：分布式系统共识算法，包含 Basic Paxos（多节点就某个值达成共识）和 Multi-Paxos（执行多次 Basic Paxos）。被公认为非常难以理解和实现。

**Raft**：2013 年诞生，是 Multi-Paxos 的变种，更易理解和工程实现。将问题分解为：领导者选举、日志复制、安全性。

## 网关

### 为什么需要网关？

微服务背景下，安全认证、流量控制、日志监控等功能每个服务都需要，网关将这些**公共功能统一管理**，主要做**请求过滤**。

### 常见网关系统

| 网关 | 特点 |
|------|------|
| **Netflix Zuul** | Netflix 开发，1.x 基于同步 IO（性能差），2.x 基于 Netty 异步 IO |
| **Spring Cloud Gateway** | Spring Cloud 生态，基于 WebFlux + Netty 异步 IO，推荐使用 |
| **Kong** | 基于 OpenResty（Nginx + Lua），高性能，插件机制扩展 |
| **APISIX** | 基于 Nginx + etcd，Apache 顶级项目，动态路由和插件热加载，性能优于 Kong |
| **Shenyu** | 基于 WebFlux，Apache 顶级项目，插件可扩展和热插拔 |

## 分布式 ID

### 什么是分布式 ID？需要满足哪些要求？

分布式系统下的全局唯一标识。基本要求：**全局唯一、高性能、高可用、方便易用**。

### 分布式 ID 方案

**数据库方案：**

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **主键自增** | 通过数据库自增主键生成 ID | 简单但并发量不大、存在单点问题 |
| **号段模式** | 批量获取 ID 存在内存中 | 减少数据库访问，滴滴 Tinyid 采用 |
| **Redis** | 通过 INCR 命令原子递增 | 性能好，有序递增 |

**算法方案：**

| 方案 | 说明 | 优缺点 |
|------|------|--------|
| **UUID** | 32 个 16 进制数字（128 位） | 全局唯一但存储大、无序、无业务含义 |
| **雪花算法** | 64 位：1 位符号 + 41 位时间戳 + 10 位机器 ID + 12 位序列号 | 有序递增、高性能，需解决时钟回拨问题 |

**开源框架：**

| 框架 | 来源 | 特点 |
|------|------|------|
| **UidGenerator** | 百度 | 基于 Snowflake 改进 |
| **Leaf** | 美团 | 支持号段模式 + Snowflake，双号段优化 |
| **Tinyid** | 滴滴 | 基于号段模式，双号段缓存 + 多 DB 支持 |

## 分布式锁

### 什么是分布式锁？

分布式系统中控制多个服务对共享资源的访问。基本要求：**互斥**（同一时刻只能一个线程持有）、**高可用**。

### 基于 Redis 实现分布式锁

**最简实现**：使用 `SETNX` 命令实现互斥，`DEL` 命令释放锁。

**释放锁**：使用 Lua 脚本通过 value 判断是否为自己的锁，保证原子性。

**设置过期时间**：避免锁无法释放。使用 `SET lockKey uniqueValue EX 3 NX`（设置值和过期时间必须是原子操作）。

**自动续期**：使用 **Redisson** 的 Watch Dog 机制，自动延长锁的过期时间。

### Redis 集群下分布式锁的可靠性

Redis 集群主从同步是异步的，主节点获取锁后未同步到从节点时宕机，新主节点会导致锁失效。

**Redlock 算法**：向多个独立 Redis 实例请求加锁，半数以上成功则获取锁。但实现复杂、性能差，存在安全隐患。实际项目中**不建议使用**，单机版 Redis + Redisson 完全够用。如果需要绝对可靠的分布式锁，可以基于 **ZooKeeper** 实现。
