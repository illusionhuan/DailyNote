## 数据库基本概念

### 什么是数据库、数据库管理系统、数据库系统、数据库管理员？

- **数据库（DB）**：信息的集合，由数据库管理系统管理的数据的集合
- **数据库管理系统（DBMS）**：操纵和管理数据库的大型软件
- **数据库系统（DBS）**：由软件、数据库和数据管理员组成
- **数据库管理员（DBA）**：负责全面管理和控制数据库系统

### 什么是元组、码、候选码、主码、外码？

- **元组**：表中的每行（每条记录）就是一个元组
- **码**：能唯一标识实体的属性，对应表中的列
- **候选码**：能唯一标识一个元组，且任何子集都不能再标识的属性组
- **主码（主键）**：从候选码中选出，一个实体集只能有一个主码
- **外码（外键）**：一个关系中的属性是另一个关系中的主码
- **主属性**：候选码中出现过的属性
- **非主属性**：不包含在任何候选码中的属性

### 主键和外键有什么区别？

| 对比项 | 主键 | 外键 |
|--------|------|------|
| 作用 | 唯一标识一个元组 | 和其他表建立联系 |
| 重复 | 不允许 | 可以有重复 |
| 空值 | 不允许 | 可以为空 |
| 数量 | 一个表只能有一个 | 一个表可以有多个 |

### 为什么不推荐使用外键与级联？

阿里巴巴开发手册规定：**不得使用外键与级联，一切外键概念必须在应用层解决**。

原因：
1. **增加复杂性**：每次 DELETE/UPDATE 都必须考虑外键约束
2. **增加额外工作**：数据库需要维护外键的一致性检查
3. **对分库分表不友好**：分库分表下外键无法生效

> 但外键也有好处（保证数据一致性、级联操作方便）。如果系统不涉及分库分表、并发量不高，可以考虑使用。

## ER 图

**E-R 图**（实体-联系图）提供表示实体类型、属性和联系的方法，用来描述现实世界的概念模型。

三种联系类型：
- **1:1**（一对一）
- **1:N**（一对多）
- **M:N**（多对多）

## 数据库范式

### 1NF（第一范式）

属性（字段）**不能再被分割**。这是所有关系型数据库的最基本要求。

### 2NF（第二范式）

在 1NF 基础上，**消除了非主属性对于码的部分函数依赖**。

关键概念：
- **函数依赖**：X 的值确定时，Y 的值必定确定（X → Y）
- **部分函数依赖**：存在 X 的真子集 X0，使得 X0 → Y
- **完全函数依赖**：非主属性完全依赖于全部关键字

### 3NF（第三范式）

在 2NF 基础上，**消除了非主属性对于码的传递函数依赖**。

**传递函数依赖**：X → Y，Y → Z，且 X 不包含 Y，Y 不确定 X，则 Z 传递函数依赖于 X。

> 符合 3NF 的设计基本解决了数据冗余、插入异常、修改异常、删除异常的问题。

## 存储过程

存储过程是**一些 SQL 语句的集合**，中间加了逻辑控制语句。

**优点**：调试通过后能稳定运行，预编译执行速度快。

**缺点**：难以调试和扩展，没有移植性，消耗数据库资源。**阿里巴巴 Java 开发手册禁止使用存储过程**。

## drop、delete 与 truncate 区别？

| 对比项 | drop | truncate | delete |
|--------|------|----------|--------|
| 作用 | 删除整张表 | 清空表数据 | 删除指定行数据 |
| 语句类型 | **DDL** | **DDL** | **DML** |
| 回滚 | 不可回滚 | 不可回滚 | **可回滚** |
| 触发器 | 不触发 | 不触发 | **触发** |
| 执行速度 | 最快 | 较快 | 较慢 |
| 自增重置 | - | 重置 | 不重置 |

> `delete` 会产生 binlog 日志便于回滚，`truncate` 不会产生日志因此更快。

## 数据库设计步骤

1. **需求分析**：分析用户的数据、功能和性能需求
2. **概念结构设计**：采用 E-R 模型设计
3. **逻辑结构设计**：将 E-R 图转换成表
4. **物理结构设计**：选择合适的存储结构和存取路径
5. **数据库实施**：编程、测试和试运行
6. **运行和维护**：系统运行与日常维护
