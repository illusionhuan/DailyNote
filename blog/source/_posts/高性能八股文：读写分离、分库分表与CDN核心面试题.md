## 读写分离

### 何为读写分离？

读写分离主要是为了将对数据库的**读写操作分散到不同的数据库节点上**，小幅提升写性能，大幅提升读性能。

一般采用**一主多从**架构：一台主数据库负责写，其他从数据库负责读。主库和从库之间进行数据同步，以保证从库中数据的准确性。

### 读写分离会带来什么问题？如何解决？

**主从同步延迟**：写完主库后，数据同步到从库需要时间，导致主从数据不一致。

**解决办法：**

1. **强制路由到主库**：必须获取最新数据的读请求交给主库处理。如 Sharding-JDBC 的 `HintManager`：
   ```java
   HintManager hintManager = HintManager.getInstance();
   hintManager.setMasterRouteOnly();
   // 继续JDBC操作
   ```
2. **延迟读取**：完成写请求后，避免立即进行读操作（如支付成功后先跳转成功页，点击返回后再读取）

### 如何实现读写分离？

**两种方式：**

1. **代理方式**：在应用和数据库中间加代理层，负责分离读写请求。中间件：MySQL Router、Atlas、Maxscale、MyCat
2. **组件方式**（推荐）：引入第三方组件处理读写请求，推荐 **Sharding-JDBC**，引入 jar 包即可使用

### 主从复制原理

MySQL **binlog**（二进制日志）记录了数据库中数据的所有变化。主从复制流程：

1. 主库将数据变化写入 binlog
2. 从库连接主库
3. 从库创建 **I/O 线程**向主库请求更新的 binlog
4. 主库创建 **binlog dump 线程**发送 binlog
5. 从库 I/O 线程将 binlog 写入 **relay log**
6. 从库的 **SQL 线程**读取 relay log 同步数据到本地

> **扩展：** 阿里开源的 **Canal** 工具底层原理也是依赖 binlog，模拟主从复制过程解析 binlog 实现数据同步。

## 分库分表

### 何为分库？何为分表？

**分库：** 将数据库中的数据分散到不同的数据库上。例如用户表和订单表分别放在不同数据库，或将用户表水平切分后放在不同数据库。

**分表：** 对单表的数据进行拆分：
- **垂直拆分**：对数据表**列**的拆分，把列较多的表拆分为多张表
- **水平拆分**：对数据表**行**的拆分，把行较多的表拆分为多张表

### 什么情况下需要分库分表？

- 单表数据达到**千万级别以上**，数据库读写速度缓慢 → **分表**
- 数据库中数据占用空间越来越大，备份时间越来越长 → **分库**
- 应用并发量太大 → **分库**

### 分库分表会带来什么问题？

1. **join 操作**：表分布在不同数据库，无法使用 join，需手动封装数据
2. **事务问题**：单个操作涉及多个数据库，数据库自带事务无法满足
3. **分布式 id**：数据库自增主键无法保证全局唯一，需要引入分布式 id
4. **成本增加**：需要 DBA 参与和更多数据库服务器

### 分库分表推荐方案

**ShardingSphere**（包括 Sharding-JDBC、Sharding-Proxy、Sharding-Sidecar）是首选方案：

- 功能完善：支持读写分离、分库分表、分布式事务、数据库治理
- 生态完善：社区活跃，文档完善，更新频繁

### 分库分表后数据怎么迁移？

1. **停机迁移**：凌晨低峰期挂维护公告，写脚本将老库数据同步到新库
2. **双写方案**（不停机）：增删改操作同时写入新库，再写脚本对比新老库数据直到一致
3. **Canal 增量迁移**：依赖 binlog，开发和维护成本较低

## CDN（内容分发网络）

### 什么是 CDN？

**CDN**（Content Delivery Network）内容分发网络，将**静态资源**（图片、视频、文档、JS、CSS、HTML）分发到多个不同地理位置机房的服务器上，实现**就近访问**，加快访问速度，减轻服务器和带宽负担。

> **注意：** CDN 主要针对静态资源，全站加速（如腾讯云 ECDN、阿里云 DCDN）既可加速静态资源又可加速动态资源。

### CDN 工作原理

**1. 静态资源如何缓存到 CDN 节点？**

- **预热**：主动将源站资源同步到 CDN 节点，用户首次请求直接从 CDN 获取
- **回源**：资源不在 CDN 节点时，CDN 节点向源站获取资源
- **刷新**：资源更新时删除 CDN 节点缓存，重新从源站获取

**2. 如何找到最合适的 CDN 节点？**

**GSLB**（Global Server Load Balance，全局负载均衡）是 CDN 的大脑，基于 DNS 实现：

1. 浏览器向 DNS 服务器发送域名请求
2. DNS 根据 CNAME 向 GSLB 发送请求
3. GSLB 返回性能最好（通常距离最近）的 CDN 节点地址
4. 浏览器直接访问指定 CDN 节点

**3. 如何防止资源被盗刷？**

- **Referer 防盗链**：根据 HTTP Referer 字段限制请求来源
- **时间戳防盗链**（推荐）：加密 URL 具有时效性，过期后无法访问。签名字符串通过对加密字符串、请求路径、过期时间进行 MD5 哈希获得

## 消息队列

消息队列在高性能系统设计中扮演重要角色，主要作用：

- **异步处理**：提高系统性能，削峰，减少响应时间
- **降低耦合性**：解耦生产者和消费者
- **流量削峰**：应对突发流量高峰

常见消息队列：RabbitMQ、RocketMQ、Kafka，各有适用场景。
