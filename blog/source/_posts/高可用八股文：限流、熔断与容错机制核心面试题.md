## 高可用系统设计指南

### 什么是高可用？可用性的判断标准是啥？

高可用描述的是一个系统在**大部分时间都是可用的**，即使在发生硬件故障或系统升级时，服务仍然可用。

一般使用**几个 9** 来评判系统可用性，比如 99.9999% 代表系统在所有运行时间中只有 0.0001% 的时间是不可用的。也可以用某功能的失败次数与总请求次数之比来衡量。

### 哪些情况会导致系统不可用？

1. 黑客攻击
2. 硬件故障（服务器损坏）
3. 并发量/用户请求量激增导致服务宕掉
4. 代码坏味道导致内存泄漏等问题
5. 架构关键角色（Nginx、数据库）突然不可用
6. 自然灾害或人为破坏

### 有哪些提高系统可用性的方法？

**1. 注重代码质量，测试严格把关**

代码质量是最重要的源头。提高代码质量的工具：Sonarqube、Arthas、阿里巴巴 Java 代码规范、IDEA 代码分析。

**2. 使用集群，减少单点故障**

如 Redis 使用集群，一台实例挂了后其他实例立即顶上。

**3. 限流**

监控应用流量的 QPS 或并发线程数，达到阈值时对流量进行控制。

**4. 超时和重试机制设置**

用户请求超过指定时间得不到响应就抛出异常。重试次数一般设为 **3 次**。

**5. 熔断机制**

自动收集依赖服务的资源使用情况和性能指标，恶化或失败次数达到阈值时迅速失败，切换到备用服务。常用框架：**Hystrix**、**Sentinel**、**Resilience4j**。

**6. 异步调用**

用户请求完成后立即返回结果，具体处理后续再做。常配合**消息队列**使用。

**7. 使用缓存**

缓存热点数据，减轻数据库压力。

**8. 其他**

- 核心应用使用更好的硬件
- 监控系统资源 + 报警设置
- 注意备份，必要时回滚
- **灰度发布**：每天只发布一部分机器，观察稳定后再继续
- 定期检查/更换硬件

## 冗余

冗余设计是保证系统和数据高可用的最常见手段。

**核心思想：** 相同服务部署多份，挂掉时快速切换到备份服务。

**典型应用：**

| 方案 | 说明 |
|------|------|
| **高可用集群** | 同一份服务部署多份，挂掉时切换到另一台 |
| **同城灾备** | 相同服务部署在同一城市的不同机房，备用服务不处理请求 |
| **异地灾备** | 相同服务部署在异地的不同机房 |
| **同城多活** | 类似同城灾备，但备用服务可以处理请求 |
| **异地多活** | 服务部署在异地不同机房，同时对外提供服务 |

> 光做好冗余还不够，必须配合**故障转移**——不可用服务快速且自动地切换到可用服务。例如 Redis 哨兵模式自动将 slave 升级为 master。

## 限流

### 何为限流？为什么要限流？

限流就是对**请求的速率进行限制**，避免瞬时大量请求击垮软件系统。软件系统的处理能力是有限的，超过其处理能力范围可能直接挂掉。

### 常见限流算法

**1. 固定窗口计数器算法**

规定单位时间处理的请求数量。例如 1 分钟只能访问 33 次，用 counter 记录当前处理请求数，达到上限后拒绝后续请求，时间窗口结束后重置。

**缺点：** 无法应对突发流量，如前 55 秒无请求，后 1 秒突然 1000 个请求。

**2. 滑动窗口计数器算法**

固定窗口的升级版，将时间按一定比例分片。例如 1 分钟分为 60 个窗口，每隔 1 秒移动一次。**格子划分越多，滚动越平滑，统计越精确。**

**3. 漏桶算法**

以任意速率流入请求，以**固定速率**流出处理。当请求超过桶容量则丢弃。实现方式：准备一个队列保存请求，定期从队列中取请求执行。

**4. 令牌桶算法**

桶中装的是令牌，请求处理前需要拿到一个令牌。根据限流大小按一定速率往桶里添加令牌，桶满则不再添加。

### 单机限流

直接使用 Google Guava 的 **RateLimiter**（基于令牌桶算法）：

```java
// 1秒放5个令牌（0.2秒放1个）
RateLimiter rateLimiter = RateLimiter.create(5);
// 还支持平滑预限流：预热时间内速率逐渐提升
RateLimiter rateLimiter = RateLimiter.create(5, 3, TimeUnit.SECONDS);
```

其他选择：**Bucket4j**（支持单机+分布式限流+监控集成）、**Resilience4j**（限流+熔断+负载保护+自动重试）。

### 分布式限流

1. **借助中间件**：Sentinel 或 Redis + Lua 实现限流逻辑
2. **网关层限流**：Spring Cloud Gateway 的 `RedisRateLimiter` 基于 Redis + Lua 实现

## 超时和重试

### 超时机制

当一个请求超过指定时间没有被处理，直接取消并抛出异常（如 504 Gateway Timeout）。

两种超时：
- **连接超时（ConnectTimeout）**：客户端与服务端建立连接的最长等待时间
- **读取超时（ReadTimeout）**：客户端等待服务端处理完请求的最长时间

**超时时间建议：**
- 读取超时：**1500ms**（较普适的值）
- 连接超时：**1000ms ~ 5000ms**

> 超时值设置太高会导致慢请求堆积，设置太低会加重系统压力。建议做成**可配置化参数**放在配置中心。

### 重试机制

配合超时机制一起使用，多次发送相同请求来避免**瞬态故障**和**偶然性故障**。

**重试次数：** 通常建议 **3 次**，并设置重试间隔（如第 1 次失败后等 1 秒，第 2 次等 2 秒，第 3 次等 3 秒）。

### 重试幂等

超时和重试需要注意保证**同一个请求没有被多次执行**。

场景：用户支付请求因网络波动延迟响应，客户端触发重试导致重复支付。

**解决办法：** 执行请求前判断是否已执行过（如购买课程前检查是否已购买）。
