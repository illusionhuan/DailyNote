## 为什么需要分库分表？

单库单表在数据量和并发量增长后会遇到瓶颈：

| 瓶颈 | 表现 | 解决方案 |
|------|------|----------|
| **数据量瓶颈** | 单表行数超过千万，查询变慢 | 分表 |
| **存储瓶颈** | 单库存储空间不足，备份时间长 | 分库 |
| **并发瓶颈** | 单库连接数和 QPS 达到上限 | 分库 |
| **写入瓶颈** | 单主库写入 TPS 达到上限 | 分库 |

**经验阈值**：
- 单表行数超过 **1000 万** 或容量超过 **10 GB**，考虑分表
- 单库 QPS 超过 **5000** 或连接数接近上限，考虑分库

## 垂直拆分 vs 水平拆分

### 垂直分库

按**业务模块**拆分数据库，每个库独立部署。

```
拆分前：单库
├── users 表
├── orders 表
├── products 表
└── payments 表

拆分后：三个库
用户库：├── users 表
订单库：├── orders 表 └── order_items 表
商品库：├── products 表 └── payments 表
```

**适用场景**：微服务架构，每个服务独占数据库。
**优点**：降低单库压力，服务解耦。
**缺点**：跨库 JOIN 无法使用，分布式事务复杂。

### 垂直分表

将**字段较多**的表拆分为主表和扩展表。

```sql
-- 拆分前：一张大表
users (id, name, email, phone, avatar, bio, address, ...)

-- 拆分后：主表 + 扩展表
users (id, name, email, phone)           -- 高频访问字段
user_profiles (user_id, avatar, bio, address, ...)  -- 低频访问字段
```

**原则**：把经常一起查询的字段放一张表，不常用的拆出去。

### 水平分表

将**行数过多**的表拆分为多张结构相同的表。

```
拆分前：orders 表（1亿行）

拆分后：
orders_0（2500万行）
orders_1（2500万行）
orders_2（2500万行）
orders_3（2500万行）
```

**查询时需要知道数据在哪张表**，这是水平分表的核心问题。

### 水平分库

将数据分散到多个数据库实例中。

```
DB_0: orders_0, orders_1
DB_1: orders_2, orders_3
```

水平分库 = 水平分表 + 多数据库实例。

## 分片策略

### 范围分片（Range Sharding）

按某个字段的范围划分。

```sql
-- 按 ID 范围
orders_0: id 1 ~ 25000000
orders_1: id 25000001 ~ 50000000
orders_2: id 50000001 ~ 75000000
orders_3: id 75000001 ~ 100000000

-- 按时间范围
orders_2025: created_at 在 2025 年
orders_2026: created_at 在 2026 年
```

**优点**：扩容方便，新增分片即可。
**缺点**：容易产生热点，新数据集中在最后一个分片。

### 哈希分片（Hash Sharding）

对分片键取哈希后取模。

```java
// 公式：hash(sharding_key) % 分片数
int shardIndex = userId.hashCode() % 4;  // 0, 1, 2, 3
String table = "orders_" + shardIndex;
```

**优点**：数据分布均匀，无热点问题。
**缺点**：扩容时需要数据迁移（取模数变化）。

### 一致性哈希分片

```
哈希环：
    0 ──── node_0 ──── node_1 ──── node_2 ───── 2^32
    │                                     │
    └─────────────────────────────────────┘

数据按哈希值顺时针找到最近的节点
```

**优点**：扩容时只迁移部分数据。
**缺点**：实现复杂，数据分布可能不均匀（需要虚拟节点）。

### 分片策略对比

| 策略 | 数据均匀性 | 扩容成本 | 范围查询 | 实现复杂度 |
|------|-----------|----------|----------|-----------|
| 范围分片 | 低（热点） | 低 | ✅ 支持 | 低 |
| 哈希分片 | 高 | 高 | ❌ 不支持 | 中 |
| 一致性哈希 | 中 | 中 | ❌ 不支持 | 高 |

**推荐**：大多数场景使用**哈希分片**，需要范围查询时配合**二级索引表**。

## 分片键选择

分片键决定了数据路由到哪个分片，选择不当会导致：

### 选择原则

1. **高频查询条件**：查询中必须包含分片键，否则需要广播所有分片
2. **高基数**：分片键的值域要足够大（如 user_id 比 status 好）
3. **数据均匀**：避免某个分片数据过多

### 常见分片键选择

| 业务场景 | 分片键 | 理由 |
|----------|--------|------|
| 用户订单查询 | user_id | 按用户查询订单是最高频操作 |
| 商品订单查询 | product_id | 按商品维度统计 |
| 多租户系统 | tenant_id | 租户数据隔离 |
| 日志系统 | created_at | 按时间归档和清理 |

### 分片键选择的反模式

```sql
-- 反模式：分片键不在查询条件中
-- 分片键是 user_id，但查询按 order_id
SELECT * FROM orders WHERE order_id = 12345;
-- 需要广播到所有分片（4个库），性能差

-- 解决方案：建立 order_id → user_id 的映射表
-- 或使用全局二级索引
```

## 分布式 ID 方案

分库分表后，数据库自增 ID 无法保证全局唯一，需要引入分布式 ID 方案。

### 方案对比

| 方案 | 原理 | 有序性 | 性能 | 依赖 |
|------|------|--------|------|------|
| UUID | 随机生成 | 无序 | 高 | 无 |
| 数据库自增 | 独立 ID 数据库 | 有序 | 低 | 数据库 |
| 号段模式 | 批量获取 ID 段 | 有序 | 高 | 数据库 |
| **雪花算法** | 时间戳+机器ID+序列号 | 趋势递增 | 很高 | 时钟 |
| Leaf（美团） | 号段+雪花双模式 | 有序 | 很高 | ZooKeeper |
| Tinyid（滴滴） | 号段模式 | 有序 | 很高 | 数据库 |

### 雪花算法（Snowflake）

```
64 位 ID 结构：
┌─ 1位符号位 ─┬─ 41位时间戳 ─┬─ 10位机器ID ─┬─ 12位序列号 ─┐
│     0      │  毫秒级时间  │  0~1023     │   0~4095    │
└────────────┴──────────────┴─────────────┴─────────────┘

- 41 位时间戳：可用 69 年
- 10 位机器 ID：支持 1024 个节点
- 12 位序列号：同一毫秒生成 4096 个 ID
- 总计：每节点每秒可生成约 400 万个 ID
```

```java
// 雪花算法核心实现
public class SnowflakeIdGenerator {
    private final long epoch = 1704067200000L; // 2024-01-01
    private final long machineBits = 10;
    private final long sequenceBits = 12;
    private final long maxMachineId = (1L << machineBits) - 1;
    private final long maxSequenceId = (1L << sequenceBits) - 1;

    private long machineId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public synchronized long nextId() {
        long currentTimestamp = System.currentTimeMillis();
        if (currentTimestamp < lastTimestamp) {
            throw new RuntimeException("时钟回拨");
        }
        if (currentTimestamp == lastTimestamp) {
            sequence = (sequence + 1) & maxSequenceId;
            if (sequence == 0) {
                currentTimestamp = waitNextMillis();
            }
        } else {
            sequence = 0;
        }
        lastTimestamp = currentTimestamp;
        return ((currentTimestamp - epoch) << (machineBits + sequenceBits))
             | (machineId << sequenceBits)
             | sequence;
    }
}
```

### 号段模式

```java
// 号段模式：每次从数据库批量获取一段 ID
// 表结构
CREATE TABLE id_generator (
    biz_tag VARCHAR(128) NOT NULL COMMENT '业务标识',
    max_id BIGINT NOT NULL DEFAULT 0 COMMENT '当前最大ID',
    step INT NOT NULL DEFAULT 1000 COMMENT '步长',
    PRIMARY KEY (biz_tag)
);

-- 获取号段
UPDATE id_generator SET max_id = max_id + step WHERE biz_tag = 'order';
-- 应用层获得 [max_id - step + 1, max_id] 这段 ID
```

**优点**：ID 有序、性能高（内存分配）、对数据库压力小（批量获取）。
**缺点**：重启后可能浪费一段 ID。

## 主流分库分表中间件

### 中间件对比

| 中间件 | 类型 | 语言 | 特点 |
|--------|------|------|------|
| **ShardingSphere-JDBC** | 客户端分片 | Java | 性能最高，无额外网络开销 |
| **ShardingSphere-Proxy** | 代理层分片 | Java | 语言无关，运维友好 |
| MyCat | 代理层分片 | Java | 社区活跃度下降 |
| Vitess | 代理层分片 | Go | YouTube 出品，K8s 原生 |
| TDDL | 客户端分片 | Java | 阿里内部使用 |

### ShardingSphere-JDBC 详解

ShardingSphere 是 Apache 顶级项目，推荐使用 **ShardingSphere-JDBC**。

```xml
<!-- Maven 依赖 -->
<dependency>
    <groupId>org.apache.shardingsphere</groupId>
    <artifactId>shardingSphere-jdbc</artifactId>
    <version>5.4.1</version>
</dependency>
```

```yaml
# 配置示例：水平分表
dataSources:
  ds_0:
    url: jdbc:mysql://localhost:3306/db_0
    username: root
    password: root
  ds_1:
    url: jdbc:mysql://localhost:3306/db_1
    username: root
    password: root

rules:
  - !SHARDING
    tables:
      orders:
        actualDataNodes: ds_${0..1}.orders_${0..3}
        tableStrategy:
          standard:
            shardingColumn: user_id
            shardingAlgorithmName: orders_inline
        keyGenerateStrategy:
          column: id
          keyGeneratorName: snowflake
    shardingAlgorithms:
      orders_inline:
        type: INLINE
        props:
          algorithm-expression: orders_${user_id % 4}
```

```java
// 使用方式与普通数据源完全一致
@Autowired
private DataSource dataSource;  // ShardingSphere 自动注入

public Order getOrderById(Long userId, Long orderId) {
    // SQL 自动路由到正确的分片
    String sql = "SELECT * FROM orders WHERE user_id = ? AND id = ?";
    // ShardingSphere 根据 user_id 自动路由
}
```

### ShardingSphere-Proxy

```bash
# 部署 Proxy（相当于一个 MySQL 代理）
docker run -d apache/shardingsphere-proxy

# 客户端连接 Proxy 像连接普通 MySQL
mysql -h proxy-host -P 3307 -u root -p
```

**适用场景**：
- 非 Java 语言的应用
- 不想修改应用代码
- 需要 DBA 通过标准 MySQL 协议管理

## 分库分表带来的问题与解决方案

### 1. 跨分片 JOIN

```sql
-- 问题：orders 和 users 在不同库
SELECT o.*, u.name FROM orders o JOIN users u ON o.user_id = u.id;

-- 解决方案1：广播表（小表全量复制到每个分片）
-- users 表数据量小，配置为广播表
rules:
  - !SHARDING
    broadcastTables:
      - users

-- 解决方案2：绑定表（相同分片键的表放在一起）
-- orders 和 order_items 都按 user_id 分片
rules:
  - !SHARDING
    bindingTables:
      - orders, order_items

-- 解决方案3：应用层组装
// 先查 orders，再用 user_id 批量查 users
List<Order> orders = orderDao.findByUserId(userId);
List<Long> userIds = orders.stream().map(Order::getUserId).collect(toList());
List<User> users = userDao.findByIds(userIds);
```

### 2. 跨分片事务

```java
// 方案1：ShardingSphere 分布式事务（XA 模式）
@Transactional
public void createOrder(Order order, Payment payment) {
    orderDao.insert(order);      // orders 库
    paymentDao.insert(payment);  // payments 库
    // ShardingSphere 自动管理 XA 事务
}

// 方案2：柔性事务（TCC 或 Seata AT 模式）
// 适用于高并发场景，最终一致性
```

### 3. 跨分片排序与分页

```sql
-- 问题：分页需要合并多个分片的结果
SELECT * FROM orders ORDER BY created_at DESC LIMIT 1000000, 10;

-- ShardingSphere 自动处理：
-- 1. 向每个分片发送 SELECT * FROM orders ORDER BY created_at DESC LIMIT 1000010, 10
-- 2. 在内存中合并排序
-- 3. 取最终的 10 条

-- 大偏移量分页仍然有性能问题，解决方案：
-- 游标分页：WHERE id > lastId ORDER BY id LIMIT 10
```

### 4. 全局唯一约束

```sql
-- 问题：唯一索引只在单分片生效
-- user_id + order_no 的唯一约束在分库分表后无法保证

-- 解决方案1：分布式锁
// 创建订单前加锁
redisLock.lock("order:" + orderNo);
try {
    orderDao.insert(order);
} finally {
    redisLock.unlock("order:" + orderNo);
}

-- 解决方案2：全局唯一索引表
CREATE TABLE unique_order_no (
    order_no VARCHAR(64) PRIMARY KEY
);
-- 先插入唯一表，成功后再插入订单表
```

### 5. 数据扩容

```sql
-- 4 个分片扩容到 8 个分片
-- 哈希分片需要重新取模，大量数据迁移

-- 方案1：停机迁移（简单但影响业务）
-- 方案2：双写迁移（推荐）
-- 1. 新旧库同时写入
-- 2. 后台任务迁移历史数据
-- 3. 数据校验一致后切换读流量
-- 4. 关闭旧库写入

-- 方案3：一致性哈希（只迁移部分数据）
```

## 分库分表实战 Checklist

| 阶段 | 检查项 |
|------|--------|
| **评估** | 单表是否真的超过千万级？先尝试优化 SQL 和索引 |
| **策略** | 选择合适的分片策略和分片键 |
| **ID** | 引入分布式 ID 方案（雪花算法或号段模式） |
| **中间件** | 推荐 ShardingSphere-JDBC |
| **绑定表** | 相同分片键的表配置为绑定表 |
| **广播表** | 小表（配置表、字典表）配置为广播表 |
| **事务** | 评估是否需要分布式事务 |
| **监控** | 部署分片监控，关注数据倾斜 |
| **扩容** | 提前规划扩容方案 |

## 何时不需要分库分表？

分库分表是**最后手段**，在以下方案无效时才考虑：

1. **SQL 优化**：EXPLAIN 分析，添加索引
2. **读写分离**：主写从读，分散压力
3. **缓存**：Redis 缓存热点数据
4. **归档**：历史数据迁移到归档表
5. **升级硬件**：SSD、更大内存、更多 CPU

**分库分表的代价很大**：开发复杂度增加、运维成本提高、跨分片查询困难。能不分就不分。
