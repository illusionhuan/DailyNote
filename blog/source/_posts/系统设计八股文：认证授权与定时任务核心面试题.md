## 认证授权

### 认证和授权的区别？

- **认证（Authentication）**：你是谁？验证用户身份的凭据（用户名/密码）
- **授权（Authorization）**：你有权限干什么？掌管访问系统的权限

认证发生在授权之前，两者结合使用保护系统安全性。

### RBAC 模型了解吗？

**RBAC（基于角色的权限访问控制）** 是系统权限控制最常用的模型。通过**角色关联权限，角色关联用户**：

- 一个用户可以拥有**若干角色**
- 每个角色可以被分配**若干权限**
- 用户与角色、角色与权限之间构成**多对多**关系

## Cookie 和 Session

### Cookie 的作用是什么？

Cookie 存放在**客户端**，一般用来保存用户信息：
1. 保存已登录用户信息，下次访问自动填充
2. 保存 SessionId 或 Token，后端通过它识别用户状态
3. 记录和分析用户行为

### Cookie 和 Session 的区别？

- **Cookie**：数据保存在**客户端**
- **Session**：数据保存在**服务端**，安全性更高

### Session-Cookie 方案

1. 用户登录后，服务器创建 Session 并存储
2. 服务器返回 SessionID 写入用户 Cookie
3. 后续请求携带 Cookie 中的 SessionID
4. 服务器比对 SessionID 验证身份

### 多服务器节点下 Session-Cookie 方案

1. **哈希策略**：特定用户请求分配到同一服务器（宕机会丢失 Session）
2. **Session 同步**：每台服务器保存全量 Session（成本太高）
3. **集中存储**（推荐）：用 Redis 等存储 Session 信息

### 无 Cookie 时 Session 还能用吗？

**能。** 可以将 SessionID 放在 URL 参数中（`https://example.com/?session_id=xxx`），但安全性和用户体验下降。

### 为什么 Cookie 无法防止 CSRF，而 Token 可以？

**CSRF（跨站请求伪造）** 利用用户的 Cookie 发送恶意请求。

- Cookie 中的 SessionId 由**浏览器自动发送**，攻击者可利用此特性
- Token 存放在 **localStorage** 中，需要**前端主动添加**到请求头，非法请求不会携带 Token

## JWT

### 什么是 JWT？

**JWT（JSON Web Token）** 是一种用于身份验证的令牌。由三部分组成：
- **Header**：算法和令牌类型
- **Payload**：用户信息和声明
- **Signature**：签名（防篡改）

### 如何防止 JWT 被篡改？

JWT 的第三部分 **Signature** 是对前两部分的签名。服务端用密钥对 Header + Payload 进行签名，如果有人篡改了 Payload，签名就会不匹配。

## SSO 和 OAuth 2.0

### 什么是 SSO？

**SSO（单点登录）**：用户登录其中一个子系统，就有权访问其他相关系统。如登录京东金融后，京东超市、京东国际等也能自动登录。

**好处**：用户一次登录多次使用；管理员只需维护统一账号中心；新系统直接对接即可。

### 什么是 OAuth 2.0？

OAuth 2.0 是一种**授权协议**，主要用于**第三方应用获取有限权限**。最终目的是为第三方应用颁发有时效性的令牌，使其能获取相关资源。

常见场景：**第三方登录**（微信/GitHub 登录）、**支付场景**（微信支付/支付宝）。

## 定时任务

### 为什么需要定时任务？

常见场景：凌晨数据备份、下单超时自动取消、定时抓取数据、定时发送文章、每日计算收益。

### 单机定时任务技术选型

| 技术 | 特点 |
|------|------|
| **Timer** | JDK 自带，单线程串行执行，异常会直接停止 |
| **ScheduledExecutorService** | 多线程并发执行，Timer 的替代品 |
| **Spring Task** | 通过 @Scheduled 注解定义，支持 Cron 表达式，底层基于 ScheduledThreadPoolExecutor |
| **时间轮** | 环形队列，任务写入和执行时间复杂度 O(1)，Kafka/Netty/ZooKeeper 都有实现 |

### 分布式定时任务技术选型

| 框架 | 特点 |
|------|------|
| **Quartz** | Java 定时任务领域老大哥，支持动态添加任务和集群，但分布式支持基于数据库锁，有弊端 |
| **Elastic-Job** | 当当开源，基于 Quartz + ZooKeeper，支持分布式分片和高可用，去中心化设计 |
| **XXL-JOB**（推荐） | 轻量级分布式任务调度框架，开箱即用，内置 UI 管理控制台，中心化设计 |
| **PowerJob** | 分布式任务调度领域新星，功能全面，稳定性待考察 |
