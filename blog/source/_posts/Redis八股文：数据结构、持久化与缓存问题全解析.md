## Redis 线程模型

### Redis 单线程模型了解吗？

Redis 的单线程指的是 **Redis 的网络 IO 和键值对读写是由一个线程来完成的**，这也是 Redis 对外提供键值存储服务的主要流程。但 Redis 的其他功能（如持久化、异步删除、集群数据同步等）是由额外的线程执行的。

### Redis 6.0 之前为什么不使用多线程？

1. **单线程编程容易**，更容易维护
2. Redis 的**性能瓶颈不在 CPU**，主要在内存和网络
3. 多线程会存在**死锁、线程上下文切换**等问题，甚至会影响性能

### Redis 6.0 之后为何引入了多线程？

Redis 6.0 引入多线程主要是为了**提高网络 IO 读写性能**。虽然引入了多线程，但 Redis 的多线程只是在**网络数据的读写**这类耗时操作上使用，**执行命令仍然是单线程顺序执行**，不需要担心线程安全问题。

Redis 6.0 的多线程默认是禁用的，开启需要修改 `redis.conf`：

```
io-threads-do-reads yes
io-threads 4  # 4核机器建议设置为2或3个线程，8核建议设置为6个线程
```

## Redis 内存管理

### Redis 给缓存数据设置过期时间有啥用？

内存是有限的，如果缓存中所有数据一直保存，会 Out of Memory。很多业务场景需要数据只在某时间段内存在，比如短信验证码（1分钟有效）、登录 token（1天有效）。

Redis 通过**过期字典（hash 表）**来保存数据过期的时间，过期字典的键指向数据库中的某个 key，值是毫秒精度的 UNIX 时间戳。

### 过期数据的删除策略

1. **惰性删除**：只在取出 key 的时候才检查是否过期。对 CPU 最友好，但可能造成过期 key 堆积
2. **定期删除**：每隔一段时间抽取一批 key 执行删除过期 key 操作。对内存更友好

Redis 采用的是 **定期删除 + 惰性删除** 两种策略配合使用。

### Redis 内存淘汰机制

仅靠过期策略可能漏掉大量过期 key，导致 OOM。Redis 提供 **8 种数据淘汰策略**：

| 策略 | 说明 |
|------|------|
| `volatile-lru` | 从已设置过期时间的数据中挑选**最近最少使用**的数据淘汰 |
| `volatile-ttl` | 从已设置过期时间的数据中挑选**将要过期**的数据淘汰 |
| `volatile-random` | 从已设置过期时间的数据中**任意选择**数据淘汰 |
| `allkeys-lru` | 当内存不足时，移除**最近最少使用**的 key（最常用） |
| `allkeys-random` | 从数据集中**任意选择**数据淘汰 |
| `no-eviction` | 禁止驱逐数据，内存不足时新写入操作报错 |
| `volatile-lfu` | 从已设置过期时间的数据中挑选**最不经常使用**的数据淘汰（4.0+） |
| `allkeys-lfu` | 当内存不足时，移除**最不经常使用**的 key（4.0+） |

## Redis 持久化机制

### RDB 持久化

Redis 可以通过创建**快照（snapshot）**来获得存储在内存中的数据在某个时间点上的副本。RDB 是 Redis **默认**的持久化方式。

```
save 900 1      # 900秒内至少1个key发生变化，触发bgsave
save 300 10     # 300秒内至少10个key发生变化
save 60 10000   # 60秒内至少10000个key发生变化
```

**RDB 创建快照时会阻塞主线程吗？**
- `save`：主线程执行，**会阻塞**
- `bgsave`：子线程执行，**不会阻塞**（默认选项）

### AOF 持久化

与快照相比，AOF 持久化的**实时性更好**，已成为主流方案。默认未开启，需配置 `appendonly yes`。

三种同步策略：

| 策略 | 说明 |
|------|------|
| `always` | 每次数据修改都写入 AOF 文件，严重降低性能 |
| `everysec` | 每秒同步一次（推荐），最多丢失1秒数据 |
| `no` | 让操作系统决定何时同步 |

**AOF 日志实现方式：** 与关系型数据库不同，Redis AOF 是在**执行完命令之后**再记录日志。好处是不阻塞当前命令执行且无需语法检查，风险是宕机可能丢失数据。

### AOF 重写

AOF 重写可以产生一个体积更小但保存相同数据库状态的新 AOF 文件。通过读取数据库中的键值对来实现，无需对现有 AOF 文件进行读入分析。

### Redis 4.0 混合持久化

Redis 4.0 开始支持 **RDB + AOF 混合持久化**（配置 `aof-use-rdb-preamble` 开启）。AOF 重写时直接把 RDB 内容写到 AOF 文件开头，结合了 RDB 快速加载和 AOF 数据安全的优点。

## Redis 事务

### 如何使用 Redis 事务？

通过 `MULTI`、`EXEC`、`DISCARD`、`WATCH` 命令实现：

```
> MULTI
> SET USER "Guide"
QUEUED
> GET USER
QUEUED
> EXEC
1) OK
2) "Guide"
```

`WATCH` 命令用于监听指定的键，如果被监视的键在事务执行前被修改，整个事务不会执行。

### Redis 支持原子性吗？

**不支持。** Redis 事务在运行错误的情况下，除了执行过程中出现错误的命令外，其他命令都能正常执行，并且 **Redis 不支持回滚**。

Redis 事务可以理解为：**将多个命令打包，按顺序执行，不会被中途打断**。

### 如何解决 Redis 事务的缺陷？

使用 **Lua 脚本**批量执行多条 Redis 命令，一次性提交到服务器执行，大幅减小网络开销。一段 Lua 脚本执行过程中不会有其他脚本或命令同时执行。

Redis 7.0 新增了 **Redis Functions** 特性，可以看作比 Lua 更强大的脚本。

## Redis 性能优化

### 什么是 bigkey？

如果一个 key 对应的 value 占用的内存比较大，就是 bigkey。参考标准：
- String 类型 value 超过 **10KB**
- 复合类型 value 包含元素超过 **5000 个**

### bigkey 有什么危害？

- 消耗更多内存空间
- 读写效率下降，阻塞其他请求
- 集群模式下数据不均衡

### 如何发现 bigkey？

1. **`redis-cli --bigkeys`**：扫描所有 key，找出每种数据结构的 top1 bigkey
2. **分析 RDB 文件**：使用 `redis-rdb-tools`（Python）或 `rdb_bigkeys`（Go）等工具

### 大量 key 集中过期问题

定期删除过程中遇到大量过期 key 时，客户端请求必须等待清理完成，导致响应变慢。

**解决办法：**
1. 给 key 设置**随机过期时间**
2. 开启 **lazy-free**（Redis 4.0+），异步方式延迟释放 key 使用的内存

## Redis 生产问题

### 缓存穿透

**定义：** 大量请求的 key 根本不存在于缓存中，请求直接打到数据库。

**解决办法：**

1. **参数校验**：不合法的参数直接抛异常
2. **缓存无效 key**：查询不到时写一个空值到 Redis 并设置过期时间
3. **布隆过滤器**：把所有可能存在的请求值存放在布隆过滤器中，不存在则直接返回

> **布隆过滤器特性：** 说某个元素存在，小概率误判；说某个元素不在，那么一定不在。

### 缓存雪崩

**定义：** 缓存同一时间大面积失效，后面的请求都直接落到数据库上。

**解决办法：**

针对 Redis 服务不可用：
1. 采用 **Redis 集群**，避免单机故障
2. **限流**，避免同时处理大量请求

针对热点缓存失效：
1. 设置**不同的失效时间**（随机值）
2. 缓存**永不失效**

### 如何保证缓存和数据库数据的一致性？

**Cache Aside Pattern（旁路缓存模式）：** 遇到写请求时，**更新 DB，然后直接删除 cache**。

如果更新数据库成功而删除缓存失败：
1. **缓存失效时间变短**（不推荐，治标不治本）
2. **增加 cache 更新重试机制**（常用）：隔一段时间重试，多次失败则存入队列，等缓存服务可用后再删除
